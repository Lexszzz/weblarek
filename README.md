# Интернет-магазин «Web-Larеk»
«Web-Larеk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`


#### Класс Api
Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter
Обработчик событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.


### Данные
Данные хранятся с помощью двух интерфейсов: товара и  покупателя.

Интерфейс товара (IProduct) описывает структуру данных, связанных с единицей каталога. Он включает в себя уникальный идентификатор, текстовое описание, ссылку на изображение, название товара, категорию товара и его цену. Поле price допускает значение null, что позволяет учитывать товары без установленной цены.

interface IProduct {
  id: string;           // Уникальный идентификатор товара
  description: string;  // Описание товара
  image: string;        // URL изображения товара
  title: string;        // Название товара
  category: string;     // Категория товара
  price: number | null; // Цена (может отсутствовать)
}

Интерфейс покупателя (IBuyer) определяет данные, необходимые для оформления заказа. Он содержит информацию о способе оплаты, а также контактные данные пользователя: электронную почту, телефон и адрес доставки.

interface IBuyer {
  payment: TPayment; // Способ оплаты (тип описывается отдельно)
  email: string;     // Email покупателя
  phone: string;     // Телефон
  address: string;   // Адрес доставки
}

### Модели данных
Для учета данных в приложении используются три класса, разделенные между собой по смыслу и зонам ответственности:

##### Каталог товаров:
хранит следующие данные:

products: LarekProduct[] //массив всех товаров
selectedProduct: LarekProduct | null //выбранный товар или его отсутствие

содержит методы:

setProducts   //сохранение массива товаров полученного в параметрах метода;

setSelectedProduct  //сохранение товара для подробного отображения;
getSelectedProduct    //получение товара для подробного отображения.

##### Корзина:
хранит следующие данные:
items: LarekProduct[] //массив товаров, выбранных покупателем для покупки

методы:
getItems  //получение массива товаров, которые находятся в корзине;
addItem   //добавление товара, который был получен в параметре, в массив корзины;
removeItem  //удаление товара, полученного в параметре из массива корзины;
clear   //очистка корзины;
getTotalPrice     //получение стоимости всех товаров в корзине;
getTotalCount     //получение количества товаров в корзине;
hasItem       //проверка наличия товара в корзине по его id, полученного в параметр метода.

##### Покупатель:
хранит следующие данные:
payment: TPayment | null //вид оплаты или его отсутствие
email: string;  //email
phone: string; //телефон
address: string; //адреc

содержит методы:
setData     //сохранение данных в модели
getData     //получение всех данных покупателя
clear     //очистка данных покупателя
validate      //валидация данных

#### Слой коммуникации

В приложении используется отдельный класс, отвечающий за взаимодействие с сервером. Данный класс инкапсулирует работу с API и использует композицию — принимает класс Api через конструктор.

Класс выполняет задачи:
larekProductsResponse //получение списка товаров с сервера (GET)
larekOrderRequest //отправка данных заказа на сервер (POST) 
larekOrderResponse //получение ответа сервера после оформления заказа

Методы класса используют интерфейс IApi и его методы GET и POST, что позволяет разделить сетевую логику и бизнес логику приложения.


#### Классы слоя представления (VIEW)


##### Класс BaseCard
Базовый класс для отображения карточек товаров. Используется как родительский класс для компонентов карточек (каталог, превью, элементы корзины). Класс реализует только общий функционал отображения названия и цены, без хранения данных.

###### Тип данных: BaseCardData
Описывает минимальный набор данных, используемый базовым классом:
type BaseCardData = {
  title: string;       // название товара
  priceLabel: string;  // строковое представление цены
};

###### Наследование
BaseCard<T extends BaseCardData> extends Component<T>
Класс является дженериком и позволяет дочерним компонентам расширять тип данных (например добавлять изображение, категорию, описание и т.д.)

###### Конструктор
constructor(container: HTMLElement)
Принимает корневой DOM-элемент карточки и находит внутри него базовые элементы разметки:
.card__title — название товара
.card__price — цена
Ссылки на элементы сохраняются в полях класса

###### Поля класса
titleElement: HTMLElement — элемент заголовка
priceElement: HTMLElement — элемент цены

###### Сеттеры
Класс использует сеттеры для обновления DOM при вызове метода render:
title: string — обновляет текст заголовка
priceLabel: string — устанавливает текстовое представление цены

###### Дополнительно
BaseCard является основой для следующих компонентов:
ProductCard — карточка в каталоге
ProductPreview — карточка в модальном окне
CartItem — элемент списка корзины


##### Класс ProductCard
Компонент отображения карточки товара в каталоге (главная страница).
Отвечает за отображение краткой информации о товаре и обработку клика по карточке.

###### Тип данных: ProductCardData
Расширяет базовый тип BaseCardData дополнительными полями, необходимыми для отображения карточки в каталоге:
type ProductCardData = BaseCardData & {
  image: string;     // ссылка на изображение
  category: string;  // категория товара
};

###### Наследование
ProductCard extends BaseCard<ProductCardData>
Наследует базовую логику отображения (title, priceLabel) и дополняет её отображением изображения и категории

###### Конструктор
constructor(container: HTMLElement, actions: ProductCardActions)
Принимает:
container: HTMLElement — корневой DOM-элемент карточки
actions: ProductCardActions — объект с обработчиками действий
type ProductCardActions = {
  onClick: () => void;
};
В конструкторе:
вызывается родительский конструктор
находятся дополнительные элементы:
.card__image — изображение
.card__category — категория
устанавливается обработчик клика на карточку через actions.onClick

###### Поля класса
imageElement: HTMLImageElement — элемент изображения
categoryElement: HTMLElement — элемент категории

###### Сеттеры
image: string - Устанавливает изображение карточки через метод setImage
category: string - Устанавливает текст категории и обновляет CSS-классы
сбрасывает предыдущие классы - Добавляет модификатор на основе categoryMap

###### Обработка событий
Компонент не генерирует события напрямую
При клике вызывается переданный обработчик:
actions.onClick()

В Presenter этот обработчик приводит к генерации события:
events.emit("product:select", { id })
Генерируемые события:
product:select — пользователь выбрал карточку товара
Данные:
{ id: string }

###### Дополнительно
Компонент не хранит id товара
Компонент не использует EventEmitter напрямую
Все действия передаются через actions

###### Сценарии логики
Клик по карточке → ProductCard (actions.onClick) → Presenter → events.emit("product:select") → Catalog.setSelectedProduct() → product:selected → ProductPreview

###### Сеттеры

CartView extends Component<CartViewData>

constructor(container: HTMLElement, events: EventEmitter)
Принимает:
container: HTMLElement — DOM-элемент корзины
events: EventEmitter — обработчик событий

сохраняется ссылка на EventEmitter

.basket__list — список товаров
.basket__price — общая стоимость

Генерируемые события
order:start — пользователь начинает оформление заказа

###### Дополнительно
Компонент не хранит данные корзины

##### Класс Header
Компонент отображения шапки сайта.
Отвечает за отображение количества товаров в корзине и обработку открытия корзины.

###### Тип данных: HeaderData
Описывает данные, необходимые для отображения:
type HeaderData = {
  counter: number; // количество товаров в корзине
};

###### Наследование
Header extends Component<HeaderData>

###### Конструктор
constructor(container: HTMLElement, events: EventEmitter)

Принимает:
container: HTMLElement — DOM-элемент шапки
events: EventEmitter — обработчик событий

В конструкторе:
сохраняется ссылка на EventEmitter
находятся элементы:
.header__basket — кнопка корзины
.header__basket-counter — счетчик
устанавливается обработчик клика по кнопке корзины

###### Поля класса
basketButton: HTMLButtonElement — кнопка открытия корзины
counterElement: HTMLElement — элемент счетчика

###### Сеттеры
counter: number - Устанавливает значение счетчика товаров в корзине

###### Обработка событий
При клике на кнопку корзины генерируется событие:
this.events.emit("basket:open");######
Генерируемые события:
basket:open — пользователь открыл корзину

###### Дополнительно
Компонент не хранит данные корзины
Получает только число товаров (counter)
Не управляет логикой — только отображает состояние

###### Сценарии логики 
cart:changed → Presenter → Header.render()
Клик по корзине → Header → basket:open → Presenter → CartView


##### Класс Modal
Компонент управления модальным окном.
Отвечает за отображение, скрытие и подстановку произвольного контента.

###### Наследование
Modal extends Component<unknown>
Компонент не использует тип данных, так как не работает с конкретной структурой — он отображает любой переданный DOM-элемент.

###### Конструктор
constructor(container: HTMLElement, events: EventEmitter)

Принимает:
container: HTMLElement — корневой DOM-элемент модального окна
events: EventEmitter — обработчик событий

В конструкторе:
сохраняется ссылка на EventEmitter
находятся элементы:
.modal__content — контейнер содержимого
.modal__close — кнопка закрытия
устанавливаются обработчики:
клик по кнопке закрытия
клик по оверлею (вне контента)

###### Поля класса
contentElement: HTMLElement — контейнер для отображаемого контента
closeButton: HTMLButtonElement — кнопка закрытия

###### Методы
setContent(content: HTMLElement): void - Очищает текущее содержимое и вставляет новый DOM-элемент
open(): void -Открывает модальное окно (добавляет класс modal_active)
close(): void - Закрывает модальное окно:
- удаляет класс modal_active
- очищает содержимое
- генерирует событие закрытия

Генерируемые события
modal:close — модальное окно закрыто

###### Обработка событий
Компонент обрабатывает:
клик по кнопке закрытия
клик по области вне контента
В обоих случаях вызывается close().

###### Дополнитеьно
Компонент не содержит бизнес-логики
Компонент не зависит от конкретных View-компонентов
Работает как универсальный контейнер
Управляется исключительно через Presenter.

###### Сценарии логики
Presenter → modal.setContent(component) Presenter → modal.open() 
Клик вне окна / крестик → Modal.close() → emit("modal:close")


##### Класс BaseForm
Базовый класс для компонентов форм.
Используется как родительский для всех форм в приложении (оформление заказа и ввод контактов).
Класс реализует общий механизм:
- обработки пользовательского ввода
- генерации событий
- отображения состояния формы (ошибки, доступность кнопки)

###### Наследование
BaseForm extends Component<Record<string, string>>

###### Конструктор
constructor(container: HTMLElement, events: EventEmitter)
Принимает:
container: HTMLElement — DOM-элемент формы
events: EventEmitter — обработчик событий
В конструкторе:
форма приводится к HTMLFormElement
находятся элементы:
.form__errors — блок ошибок
button[type="submit"] — кнопка отправки
устанавливаются обработчики:
input — изменение значения поля
submit — отправка формы

###### Поля класса
form: HTMLFormElement — корневой элемент формы
errorsElement: HTMLElement — блок ошибок
submitButton: HTMLButtonElement — кнопка отправки

###### Сеттеры
errors: string - Устанавливает текст ошибок
isValid: boolean - Управляет состоянием кнопки отправки:
true — кнопка активна
false — отключена

###### Обработка событий
Ввод данных
При изменении поля формы:
this.events.emit("form:change", {
  field: target.name,
  value: target.value,
});
Генерируется событие:
form:change — изменение одного поля формы
Данные:
{
  field: string;
  value: string;
}
Отправка формы
При отправке формы:
this.events.emit("form:submit");
Генерируется событие:
form:submit — отправка формы
Данные формы не передаются

###### Дополнительно
Класс не хранит данные формы
Класс не собирает все данные формы целиком
Presenter обновляет форму через сеттеры

###### Сценарии логики
Ввод данных→ BaseForm→ form:change { field, value }→ Presenter → Buyer.setData() 
buyer:changed → Presenter → обновление формы (setters)
submit → form:submit → Presenter → следующий шаг / отправка заказа


##### Класс OrderForm
Компонент формы первого шага оформления заказа.
Отвечает за выбор способа оплаты и ввод адреса доставки.

###### Наследование
OrderForm extends BaseForm
Наследует общий функционал работы с формами:
- обработку ввода
- генерацию событий
- управление кнопкой и ошибками

###### Конструктор
constructor(container: HTMLElement, events: EventEmitter)

Принимает:
container: HTMLElement — DOM-элемент формы
events: EventEmitter — обработчик событий

В конструкторе:
вызывается родительский конструктор
находятся элементы:
кнопки выбора оплаты (.order__buttons .button)
поле ввода адреса (input[name="address"])
для кнопок оплаты устанавливается обработчик клика
При клике:
this.events.emit("form:change", {
  field: "payment",
  value: button.name,
});

###### Поля класса
paymentButtons: HTMLButtonElement[] — кнопки выбора оплаты
addressInput: HTMLInputElement — поле ввода адреса

###### Сеттеры
address: string
Устанавливает значение поля адреса
payment: string
Обновляет выбранный способ оплаты:
добавляет класс button_alt-active для выбранной кнопки и убирает класс у остальных

###### Обработка событий
Выбор способа оплаты:
form:change { field: "payment", value: string }
Ввод адреса:
Обрабатывается в BaseForm:
form:change { field: "address", value: string }
Отправка формы:
form:submit
Генерируемые события:
form:change — изменение одного поля формы
form:submit — переход ко второму шагу оформления

###### Дополнительно
Компонент не хранит данные формы
Компонент не управляет своим состоянием напрямую

###### Сценарии логики
order:start → Presenter → OrderForm.render()
Клик по оплате / ввод адреса → form:change → Presenter → Buyer.setData()
buyer:changed → Presenter → OrderForm (setters)
submit → form:submit → Presenter → ContactsForm

#### События приложения
В приложении используется событийно-ориентированный подход.
Все взаимодействия между слоями происходят через EventEmitter.

###### События моделей данных
catalog:changed - Изменение списка товаров в каталоге
Источник: Catalog, вызывается при загрузке или обновлении списка товаров

product:selected - Изменение выбранного товара
Источник: Catalog, вызывается при выборе товара для просмотра

cart:changed - Изменение содержимого корзины
Источник: Cart, вызывается при добавлении или удалении товара, очистке корзины

buyer:changed -Изменение данных покупателя
Источник: Buyer, вызывается при обновлении данных формы

###### События приложения
В приложении используется событийно-ориентированный подход.
Все взаимодействия между слоями происходят через EventEmitter.

События генерируются слоями Model и View, а обрабатываются в Presenter (main.ts).

События моделей данных:
catalog:changed - Изменение списка товаров в каталоге
Источник: Catalog
Вызывается при загрузке или обновлении списка товаров

product:selected - Изменение выбранного товара
Источник: Catalog
Вызывается при выборе товара для просмотра

cart:changed - Изменение содержимого корзины
Источник: Cart
Вызывается при:
- добавлении товара
- удалении товара
- очистке корзины

buyer:changed - Изменение данных покупателя
Источник: Buyer
Вызывается при обновлении данных формы

###### События представления (View)
product:select - Выбор карточки товара
Источник: ProductCard (через actions)
Данные:
{ id: string }

product:toggle - Добавление или удаление товара из корзины
Источник: ProductPreview (через actions)

basket:open - Открытие корзины
Источник: Header

cart:remove - Удаление товара из корзины
Источник: CartItem (через actions)
Данные:
{ id: string }
order:start

Начало оформления заказа - Источник: CartView
form:change
Изменение одного поля формы
Источник: BaseForm
Данные:
{
  field: string;
  value: string;
}

form:submit - Отправка формы
Источник: BaseForm
Данные:
не передаются

success:close - Закрытие окна успешного оформления
Источник: Success

modal:close - Закрытие модального окна
Источник: Modal

###### Особенности архитектуры событий
1. View не является источником данных

Компоненты представления:
- не хранят данные
- не передают модели напрямую
- сообщают о действиях пользователя

2. Actions вместо прямых событий

В некоторых компонентах (ProductCard, ProductPreview, CartItem)
события вызываются через переданные обработчики (actions), а не напрямую через EventEmitter.

3. Presenter — центр управления

main.ts выполняет роль Presenter:
- принимает события
- обновляет модели
- вызывает методы View
- управляет отображением

4. Поток данных
Пользователь → View → событие → Presenter → Model (обновление данных) → Model → событие (changed) → Presenter → обновление View
